# DESIGN.md

> 本文档定义本项目的设计系统、视觉规范、交互规范、可访问性要求与设计到开发的映射规则。默认对齐Fluent2 Design System，并根据项目实际需要做轻量扩展。除非有明确例外，新增页面、组件和交互都应遵守本文档。

------

## 1. 文档目标

### 1.1 目标

本设计文档用于统一以下内容：

- 产品体验原则：产品应该给用户怎样的感知与使用效率。
- 视觉语言：颜色、字体、间距、圆角、阴影、图标、动效等基础规则。
- 组件规则：按钮、输入框、导航、表格、对话框、提示等常用UI组件的使用边界。
- 交互模式：加载、空状态、错误、确认、撤销、批量操作等模式。
- 可访问性：键盘操作、焦点顺序、语义结构、颜色对比、屏幕阅读器支持。
- 设计到开发映射：设计token、CSS变量、组件命名、状态命名、验收标准。

### 1.2 非目标

本文档不替代：

- 产品需求文档：业务流程、需求优先级、用户故事应放在PRD中。
- 技术架构文档：服务端、数据库、部署、安全架构应放在ARCHITECTURE.md中。
- 代码贡献规范：分支策略、提交规范、CI流程应放在CONTRIBUTING.md中。
- 接口文档：API协议应放在API.md、OpenAPI或对应接口平台中。

### 1.3 适用范围

本文档默认适用于：

- Web管理端
- Web应用前台
- 桌面端WebView应用
- 移动端H5页面
- 后续可能扩展到Flutter、原生移动端或小程序的设计规范

若某平台存在原生限制，应优先保持用户体验一致，再在实现细节上适配平台能力。

## 2. 设计原则

### 2.1 清晰优先

界面应先让用户理解“我在哪里、能做什么、下一步会发生什么”。视觉装饰不得优先于信息表达。

要求：

- 页面标题必须准确表达当前页面或任务。
- 主要操作必须明确，避免多个同等强度的主按钮同时出现。
- 错误信息必须说明问题原因和可执行的修复方式。
- 状态变化必须可见，例如保存中、保存成功、保存失败。

### 2.2 一致性优先

相同类型的信息、控件和交互应在不同页面中保持一致。

要求：

- 相同语义使用同一个组件。
- 相同状态使用同一种颜色、图标、文案模式。
- 相同层级使用同一套字号、间距和布局密度。
- 不为局部页面随意创建一次性样式。

### 2.3 效率优先

界面应帮助用户更快完成任务，减少不必要的判断、等待和重复输入。

要求：

- 高频操作应在主路径上可见。
- 低频或危险操作应收纳到次级入口。
- 表单应提供合理默认值、自动补全和即时校验。
- 列表应支持搜索、筛选、排序、分页或虚拟滚动。

### 2.4 包容性优先

界面必须对键盘用户、屏幕阅读器用户、低视力用户、色弱用户和不同设备用户可用。

要求：

- 任何鼠标可完成的关键操作，也应可通过键盘完成。
- 不仅依赖颜色表达状态，必须结合文本、图标或形状。
- 交互元素需要有明确的焦点态。
- 文本与背景应满足对比度要求。

### 2.5 克制表达

视觉应服务信息层级，而不是制造噪声。

要求：

- 少用高饱和度大面积色块。
- 少用不必要的阴影、渐变和动画。
- 品牌色主要用于强调关键行动、选中态和品牌识别。
- 页面密度应根据任务场景调整，不能盲目追求“炫”或“满”。

## 3. 信息架构

### 3.1 页面层级

页面按以下层级组织：

```text
应用 Shell
└── 一级模块
    └── 二级页面
        └── 内容区域
            ├── 页面标题区
            ├── 工具栏
            ├── 主内容
            ├── 辅助内容
            └── 反馈/状态区域
```

### 3.2 页面类型

| 页面类型    | 用途                   | 典型结构                            |
| ----------- | ---------------------- | ----------------------------------- |
| Dashboard   | 汇总关键指标和待办事项 | 指标卡片、图表、快捷入口、最近活动  |
| List        | 管理对象集合           | 标题、搜索、筛选、表格/卡片、分页   |
| Detail      | 查看单个对象           | 标题区、元信息、分区内容、相关操作  |
| Form        | 创建或编辑对象         | 表单分组、校验、提交、取消          |
| Workflow    | 多步骤任务             | 步骤条、当前步骤内容、上一步/下一步 |
| Settings    | 配置项管理             | 左侧分类、右侧表单、保存反馈        |
| Empty/Error | 异常路径               | 插图/图标、说明、恢复操作           |

### 3.3 导航规则

- 一级导航用于稳定业务模块，不放临时任务。
- 二级导航用于模块内分类，不超过7项为宜。
- 面包屑用于层级较深的对象详情，不用于浅层页面。
- 当前页面必须有可感知的选中态。
- 返回操作应回到用户的上一个上下文，而不是机械跳转到固定列表页。

## 4. Layout布局系统

### 4.1 基础网格

采用4px基础网格。所有间距、尺寸、圆角、图标容器应尽量落在4的倍数上；为适配图标视觉对齐，可使用2px、6px、10px等补偿值。

### 4.2 页面宽度

| 场景      | 建议宽度      |
| --------- | ------------- |
| 移动端    | 320px-767px   |
| 平板/小屏 | 768px-1023px  |
| 常规桌面  | 1024px-1439px |
| 大屏桌面  | 1440px-1919px |
| 超宽屏    | 1920px及以上  |

页面主内容不应在超宽屏上无限拉伸。阅读型内容建议限制最大宽度；数据密集型内容可占满可用区域，但应保留边距和列宽控制。

### 4.3 常用布局

#### 4.3.1 应用Shell

```text
┌──────────────────────────────────────────────┐
│ Header                                       │
├──────────────┬───────────────────────────────┤
│ Sidebar      │ Page content                  │
│              │                               │
└──────────────┴───────────────────────────────┘
```

规则：

- Header用于全局搜索、用户菜单、通知、全局操作。
- Sidebar用于一级或二级导航。
- 内容区域保留至少24px页面边距；移动端可降至16px。
- Sidebar可折叠，但折叠后图标必须可理解，必要时提供tooltip。

#### 4.3.2 页面标题区

```text
Title
Description / Metadata
Primary action + Secondary actions
```

规则：

- 标题不超过一行，必要时截断并提供完整tooltip。
- 描述用于解释页面用途，不重复标题。
- 主操作放右侧或标题区下方，保持页面内一致。

#### 4.3.3 表单布局

- 简单表单使用单列布局。
- 信息密度较高的管理端表单可使用双列，但相关字段必须就近排列。
- 标签与输入框的关系必须明确。
- 错误信息紧贴对应字段。
- 页面底部固定操作栏仅用于长表单。

### 4.4 间距Token

| Token          | 值   | 用途                 |
| -------------- | ---- | -------------------- |
| `spacing.none` | 0    | 无间距               |
| `spacing.xxs`  | 2px  | 细微视觉补偿         |
| `spacing.xs`   | 4px  | 图标与文字极近关系   |
| `spacing.s`    | 8px  | 控件内部、紧密元素   |
| `spacing.m`    | 12px | 表单字段内部、列表项 |
| `spacing.l`    | 16px | 常规组件间距         |
| `spacing.xl`   | 20px | 组件组之间           |
| `spacing.2xl`  | 24px | 页面区块间距         |
| `spacing.3xl`  | 32px | 大区块分隔           |
| `spacing.4xl`  | 40px | 页面级分隔           |
| `spacing.5xl`  | 48px | 重点区域留白         |
| `spacing.6xl`  | 56px | 大屏页面分组         |

## 5. Design Tokens

### 5.1 Token分层

采用两层token：

1. Global tokens：存储原始值，例如颜色HEX、字号、圆角、阴影、动画时长。
2. Alias tokens：表达语义用途，例如文本主色、页面背景、危险按钮背景、边框弱色。

禁止在业务组件中硬编码颜色、字号、阴影、圆角和间距。业务代码应优先使用alias token。

### 5.2 Token命名规范

命名格式：

```text
<category>.<role>.<state?>.<emphasis?>
```

示例：

```text
color.text.primary
color.text.secondary
color.background.canvas
color.background.card
color.border.default
color.status.error.foreground
radius.medium
shadow.card
motion.duration.fast
```

### 5.3 Token使用优先级

1. 优先使用组件内置token或主题变量。
2. 其次使用项目alias token。
3. 最后才使用global token。
4. 禁止在业务样式中直接写`#ffffff`、`16px`、`0 4px 12px`这类散落值，除非已记录为例外。

## 6. Color颜色系统

### 6.1 颜色角色

颜色按语义使用，不按视觉偏好使用。

| 角色               | 用途                           |
| ------------------ | ------------------------------ |
| Neutral            | 文本、背景、边框、分隔、禁用态 |
| Brand              | 主要操作、选中态、品牌识别     |
| Status             | 成功、警告、错误、信息提示     |
| Overlay            | 遮罩、浮层、弹窗背景           |
| Data Visualization | 图表、标签、分类数据           |

### 6.2 文本颜色

| Token                  | 用途                 |
| ---------------------- | -------------------- |
| `color.text.primary`   | 正文、标题、主要信息 |
| `color.text.secondary` | 辅助说明、元信息     |
| `color.text.tertiary`  | 弱提示、占位性说明   |
| `color.text.disabled`  | 禁用文本             |
| `color.text.inverse`   | 深色背景上的文本     |
| `color.text.brand`     | 品牌强调文本         |
| `color.text.error`     | 错误文本             |
| `color.text.success`   | 成功文本             |
| `color.text.warning`   | 警告文本             |

规则：

- 正文文本必须优先使用`color.text.primary`。
- 弱化信息使用`secondary`或`tertiary`，不要降低透明度制造不可控对比度。
- 错误、成功、警告文本必须有明确语义，不能仅用于装饰。

### 6.3 背景颜色

| Token                       | 用途                   |
| --------------------------- | ---------------------- |
| `color.background.canvas`   | 应用最底层背景         |
| `color.background.card`     | 卡片、面板、内容容器   |
| `color.background.subtle`   | 轻微区分的区域背景     |
| `color.background.raised`   | 浮层、菜单、弹出内容   |
| `color.background.brand`    | 品牌主按钮、强强调区域 |
| `color.background.disabled` | 禁用控件背景           |
| `color.background.overlay`  | 模态遮罩               |

### 6.4 边框颜色

| Token                   | 用途               |
| ----------------------- | ------------------ |
| `color.border.subtle`   | 轻量分隔           |
| `color.border.default`  | 常规控件边框       |
| `color.border.strong`   | 强调分隔、焦点辅助 |
| `color.border.focus`    | 焦点环             |
| `color.border.error`    | 错误字段边框       |
| `color.border.disabled` | 禁用控件边框       |

### 6.5 状态颜色

| 状态    | 前景                              | 背景                              | 使用场景                 |
| ------- | --------------------------------- | --------------------------------- | ------------------------ |
| Info    | `color.status.info.foreground`    | `color.status.info.background`    | 普通信息提示             |
| Success | `color.status.success.foreground` | `color.status.success.background` | 完成、通过、启用成功     |
| Warning | `color.status.warning.foreground` | `color.status.warning.background` | 风险、待确认、可恢复问题 |
| Error   | `color.status.error.foreground`   | `color.status.error.background`   | 失败、阻断、危险操作     |

规则：

- 危险操作使用Error语义，不使用Brand色。
- Warning不用于普通强调，只用于存在潜在风险但未失败的情况。
- Success不应大面积使用，避免造成视觉疲劳。

### 6.6 颜色可访问性

- 普通文本与背景对比度不低于4.5:1。
- 大号文本与背景对比度不低于3:1。
- 交互控件边界、焦点、图标应具备足够对比度。
- 不使用颜色作为唯一信息载体。例如错误字段还应有错误文案和图标。
- 深色模式不是简单颜色反转，应重新检查文本、边框、阴影和状态色。

## 7. Typography字体系统

### 7.1 字体族

默认字体栈：

```css
font-family: system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", "Microsoft YaHei", "PingFang SC", "Noto Sans CJK SC", sans-serif;
```

代码字体栈：

```css
font-family: "Cascadia Code", "JetBrains Mono", "SFMono-Regular", Consolas, "Liberation Mono", Menlo, monospace;
```

### 7.2 字体层级

| Token              | 字号 | 行高 | 字重 | 用途                     |
| ------------------ | ---- | ---- | ---- | ------------------------ |
| `type.display`     | 68px | 92px | 600  | 营销页超大标题，谨慎使用 |
| `type.largeTitle`  | 40px | 52px | 600  | 关键落地页标题           |
| `type.title1`      | 28px | 36px | 600  | 页面主标题               |
| `type.title2`      | 24px | 32px | 600  | 主要分区标题             |
| `type.title3`      | 20px | 28px | 600  | 卡片标题、弹窗标题       |
| `type.subtitle1`   | 18px | 24px | 600  | 强调副标题               |
| `type.subtitle2`   | 16px | 22px | 600  | 小分区标题               |
| `type.body1`       | 14px | 20px | 400  | 默认正文                 |
| `type.body1Strong` | 14px | 20px | 600  | 强调正文                 |
| `type.caption1`    | 12px | 16px | 400  | 辅助信息、元信息         |
| `type.caption2`    | 10px | 14px | 400  | 极弱信息，谨慎使用       |

### 7.3 排版规则

- 页面只能有一个视觉主标题。
- 标题应简短明确，避免整句口号化。
- 长正文优先左对齐；中文界面不建议大段居中。
- 数字、状态和短标签可使用等宽字体或tabular numbers以提升扫描效率。
- 标题和正文的层级差异应通过字号、字重和间距共同表达。
- 不通过任意加粗、变色制造临时层级。

### 7.4 文案大小写与标点

中文界面：

- 按钮文案使用动宾结构，例如“创建项目”“保存更改”。
- 表单标签使用名词或短语，例如“项目名称”“负责人”。
- 错误文案说明原因和修复方式，例如“项目名称不能为空，请输入1-40个字符”。
- 避免模糊文案，例如“操作失败”“参数错误”。

英文界面：

- 标题使用sentence case。
- 按钮使用动词开头，例如“Create project”。
- 避免全大写按钮。

## 8. Icon图标系统

### 8.1 图标原则

- 图标用于辅助识别，不替代关键文本。
- 常见动作优先使用用户熟悉的图标，例如搜索、设置、删除、下载。
- 抽象业务概念不要强行使用图标。
- 同一页面内图标风格必须一致。

### 8.2 图标尺寸

| 尺寸 | 用途                     |
| ---- | ------------------------ |
| 12px | 极小辅助标记             |
| 16px | 表格、菜单、输入框内图标 |
| 20px | 常规按钮、导航项         |
| 24px | 页面级操作、卡片强调     |
| 32px | 空状态、结果状态         |
| 48px | 大型空状态、引导页       |

### 8.3 图标可访问性

- 纯装饰图标应对屏幕阅读器隐藏。
- 有交互功能的图标按钮必须提供可访问名称。
- 图标按钮必须有tooltip或文本说明，除非含义极其明确且上下文充分。

## 9. Shape圆角与边框

### 9.1 圆角Token

| Token             | 值     | 用途                     |
| ----------------- | ------ | ------------------------ |
| `radius.none`     | 0      | 无圆角，表格分隔等       |
| `radius.small`    | 2px    | 小标签、小控件           |
| `radius.medium`   | 4px    | 输入框、按钮、菜单项     |
| `radius.large`    | 8px    | 卡片、面板、弹窗         |
| `radius.xlarge`   | 12px   | 大卡片、复杂容器         |
| `radius.circular` | 9999px | 胶囊按钮、头像、圆形控件 |

### 9.2 边框规则

- 常规边框宽度为1px。
- 强调或焦点边框可为2px，但不得造成布局跳动。
- 分隔线应弱化，不应比内容更抢眼。
- 卡片可以使用边框、阴影或背景区分，但不要三者同时强烈使用。

## 10. Elevation阴影与层级

### 10.1 使用原则

Elevation用于表达元素在z轴上的层级关系，帮助用户理解浮层、菜单、弹窗和重点内容。

不要为了“高级感”滥用阴影。阴影越强，表示越接近用户，也越应有明确交互意义。

### 10.2 阴影Token

| Token         | 用途                    |
| ------------- | ----------------------- |
| `shadow.none` | 无阴影                  |
| `shadow.2`    | 轻微浮起，例如hover卡片 |
| `shadow.4`    | 菜单、下拉、轻浮层      |
| `shadow.8`    | Popover、Tooltip        |
| `shadow.16`   | Dialog、Drawer          |
| `shadow.28`   | 关键模态层，谨慎使用    |

### 10.3 Z-index层级

| Token        | 建议值 | 用途                 |
| ------------ | ------ | -------------------- |
| `z.base`     | 0      | 普通内容             |
| `z.sticky`   | 100    | 粘性标题、固定工具栏 |
| `z.dropdown` | 1000   | 下拉菜单             |
| `z.popover`  | 1100   | Popover、Tooltip     |
| `z.overlay`  | 1200   | 遮罩                 |
| `z.drawer`   | 1300   | 抽屉                 |
| `z.dialog`   | 1400   | 模态对话框           |
| `z.toast`    | 1500   | 全局通知             |

## 11. Motion动效系统

### 11.1 动效原则

- 动效用于解释状态变化、空间关系和因果关系。
- 动效应短、自然、可中断。
- 不用闪烁、强震动、大幅移动吸引注意。
- 必须支持减少动态效果设置。

### 11.2 时长Token

| Token                       | 值    | 用途                     |
| --------------------------- | ----- | ------------------------ |
| `motion.duration.ultraFast` | 50ms  | 即时反馈                 |
| `motion.duration.fast`      | 100ms | Hover、Press             |
| `motion.duration.normal`    | 200ms | 菜单、Popover、切换      |
| `motion.duration.slow`      | 300ms | Dialog、Drawer           |
| `motion.duration.ultraSlow` | 500ms | 大范围页面过渡，谨慎使用 |

### 11.3 缓动Token

| Token                     | 用途             |
| ------------------------- | ---------------- |
| `motion.curve.accelerate` | 元素离开         |
| `motion.curve.decelerate` | 元素进入         |
| `motion.curve.ease`       | 常规状态变化     |
| `motion.curve.linear`     | 进度条、持续运动 |

### 11.4 减少动态效果

实现要求：

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    scroll-behavior: auto !important;
    transition-duration: 0.01ms !important;
  }
}
```

## 12. Component组件规范

### 12.1 组件设计原则

每个组件必须明确：

- 适用场景
- 不适用场景
- 视觉结构
- 尺寸变体
- 状态集合
- 键盘交互
- 可访问性属性
- 内容规则
- 开发API映射

### 12.2 Button按钮

#### 类型

| 类型        | 用途                               |
| ----------- | ---------------------------------- |
| Primary     | 页面主操作，一个区域内通常只放一个 |
| Secondary   | 次要操作，例如取消、返回、查看     |
| Subtle      | 低强调操作，例如工具栏操作         |
| Transparent | 极低强调，通常用于紧凑区域         |
| Danger      | 删除、清空、撤销授权等危险操作     |
| Icon Button | 空间受限或图标含义明确的操作       |

#### 状态

- Rest
- Hover
- Pressed
- Focused
- Disabled
- Loading

#### 规则

- 主按钮文案必须是明确动作，例如“保存更改”。
- 危险按钮必须明确表达后果，例如“删除项目”。
- Loading按钮应禁用重复提交，并保留按钮宽度避免抖动。
- Icon Button必须有`aria-label`或等效可访问名称。

### 12.3 Input输入框

#### 结构

```text
Label
Input + optional prefix/suffix
Hint / Error message
```

#### 规则

- 必填字段必须明确标识，但不要只用红色星号表达。
- Placeholder不能替代Label。
- 错误提示必须紧贴字段。
- 校验应尽量即时，但避免在用户输入第一个字符前立即报错。
- 密码输入必须支持显示/隐藏切换。

### 12.4 Select选择器

适用于从有限选项中选择一个或多个值。

规则：

- 少于5项且选项短，可考虑Radio而不是Select。
- 选项很多时应支持搜索。
- 多选时应展示已选数量，并支持清除。
- 禁止把复杂表单嵌入Select下拉层。

### 12.5 Checkbox复选框

适用于多选或开关式确认。

规则：

- Checkbox标签应描述选择后状态。
- 批量选择需要支持indeterminate状态。
- 用户协议确认类文案必须可读，不应过长。

### 12.6 Radio单选框

适用于少量互斥选项。

规则：

- 推荐2-5项。
- 默认应有选中值，除非业务确实允许未选择。
- 选项文案应平行一致。

### 12.7 Switch开关

适用于即时生效的二元状态。

规则：

- Switch不应用于需要提交后才生效的设置。
- 复杂后果的开关应提供说明或确认。
- Label应描述开启状态，例如“启用通知”。

### 12.8 Table表格

适用于结构化数据管理和比较。

#### 必备能力

- 列标题
- 排序
- 筛选
- 搜索
- 分页或虚拟滚动
- 空状态
- 加载状态
- 错误状态
- 行操作

#### 规则

- 表头固定适用于长表格。
- 数字右对齐，文本左对齐，状态居中或左对齐但需一致。
- 行操作过多时收纳到More菜单。
- 危险操作不得直接裸露在行内主位置。
- 批量操作必须显示已选数量。

### 12.9 Card卡片

适用于聚合少量相关信息。

规则：

- 卡片应有明确标题或主信息。
- 卡片整体可点击时，内部不应包含过多独立点击目标。
- 同一组卡片高度、内容结构和操作位置应保持一致。

### 12.10 Dialog对话框

适用于需要用户处理后才能继续当前流程的任务。

规则：

- 标题说明任务或后果。
- 正文解释必要上下文。
- 主要按钮放执行动作，次要按钮放取消。
- 危险确认必须使用Danger按钮。
- 打开Dialog后焦点进入Dialog，关闭后焦点回到触发元素。
- Esc关闭仅适用于非危险、可取消对话框。

### 12.11 Drawer抽屉

适用于不中断主上下文的详情查看、筛选、编辑。

规则：

- 右侧Drawer用于详情和编辑。
- 左侧Drawer用于导航类内容。
- Drawer内长表单应提供固定操作栏。
- 不要在Drawer内再打开复杂Drawer，避免层级混乱。

### 12.12 Tooltip提示

适用于解释短文本、图标或不可见细节。

规则：

- Tooltip只放短说明，不放关键流程信息。
- 不应包含复杂交互控件。
- 移动端需考虑无hover环境。

### 12.13 Toast通知

适用于非阻断的全局反馈。

规则：

- 成功Toast自动消失。
- 错误Toast应允许用户查看或重试。
- 关键错误不能只用Toast，页面内也应保留错误状态。
- 同一时间Toast数量应受控，避免堆叠过多。

### 12.14 Message Bar消息条

适用于页面内上下文提示。

规则：

- 放在与问题最相关的位置。
- 可关闭消息关闭后不应影响用户完成任务。
- 错误消息条应说明修复方式。

### 12.15 Tabs标签页

适用于同一上下文内的平级内容切换。

规则：

- Tab数量不宜过多，超过7项考虑导航或筛选。
- Tab切换不应丢失未保存内容。
- 当前Tab必须有明确选中态。

### 12.16 Breadcrumb面包屑

适用于多层级页面定位。

规则：

- 最后一项表示当前页面，不可点击或弱化点击。
- 面包屑不替代页面标题。
- 层级过深时中间项可折叠。

### 12.17 Menu菜单

适用于收纳命令。

规则：

- 菜单项使用动词或明确名词。
- 危险项应分组并使用危险语义。
- 禁用菜单项应说明不可用原因，必要时用tooltip。
- 菜单支持键盘方向键导航。

## 13. Interaction交互模式

### 13.1 状态模型

所有异步区域至少覆盖以下状态：

```text
idle -> loading -> success
              -> empty
              -> error
```

### 13.2 Loading加载

| 场景     | 推荐模式                       |
| -------- | ------------------------------ |
| <300ms   | 可不展示加载态，避免闪烁       |
| 300ms-1s | 显示局部spinner或按钮loading   |
| >1s      | 显示骨架屏或明确加载说明       |
| 长任务   | 显示进度、可取消或后台处理说明 |

规则：

- 优先使用局部加载，不要整页遮罩。
- 表格加载可使用Skeleton保留布局稳定。
- 按钮提交中应防止重复点击。

### 13.3 Empty空状态

空状态应说明：

1. 当前没有什么。
2. 为什么没有。
3. 用户可以做什么。

示例：

```text
暂无项目
创建第一个项目后，你可以在这里查看状态、成员和最近活动。
[创建项目]
```

### 13.4 Error错误状态

错误状态应包括：

- 问题描述
- 可能原因
- 修复动作
- 必要时的错误代码

示例：

```text
无法加载项目列表
网络连接不稳定或服务器暂时不可用。请稍后重试。
[重试]
错误代码：PROJECT_LIST_FETCH_FAILED
```

### 13.5 Success成功反馈

- 小操作成功使用Toast。
- 页面级任务成功可跳转到详情页或展示成功状态。
- 用户创建对象后，应明确下一步可做什么。

### 13.6 Confirm确认

以下操作必须确认：

- 删除不可恢复数据
- 清空大量内容
- 撤销授权
- 修改安全配置
- 批量影响多个对象

确认文案必须具体说明对象、范围和后果。

### 13.7 Undo撤销

可恢复操作优先使用Undo而不是阻断确认。

适合Undo的场景：

- 归档
- 移动
- 标记状态
- 删除但可在短时间内恢复

不适合Undo的场景：

- 永久删除
- 支付
- 授权变更
- 安全策略变更

## 14. Forms表单规范

### 14.1 字段组织

- 相关字段分组展示。
- 每组提供清晰标题。
- 复杂字段提供说明文本。
- 高风险字段提供确认或预览。

### 14.2 校验规则

| 校验时机   | 用途                       |

- 提交中禁用提交按钮。
- 成功后给出明确反馈。
- 失败后保留用户输入。
- 离开有未保存内容的页面时应提醒用户。

## 16. Accessibility可访问性

### 16.1 基础要求

- 满足WCAG 2.1 AA作为最低目标。
- 页面结构使用正确HTML语义。
- 可交互元素必须可聚焦。
- 焦点顺序与视觉顺序一致。
- 焦点态必须清晰可见。
- 颜色对比度满足要求。
- 表单字段与Label正确关联。
- 异步状态变化应被辅助技术感知。

### 16.2 键盘操作

| 控件        | 键盘行为                                    |
| ----------- | ------------------------------------------- |
| Button      | `Tab`聚焦，`Enter`/`Space`触发              |
| Link        | `Tab`聚焦，`Enter`打开                      |
| Menu        | 方向键移动，`Enter`选择，`Esc`关闭          |
| Dialog      | 焦点困在Dialog内，`Esc`按规则关闭           |
| Tabs        | 方向键切换焦点，`Enter`或自动激活按实现规范 |
| Checkbox    | `Space`切换                                 |
| Radio Group | 方向键切换选项                              |
| Select      | 方向键浏览，`Enter`选择，`Esc`关闭          |

### 16.3 ARIA使用规则

- 优先使用原生语义元素，不滥用ARIA。
- 只有在原生语义不足时使用ARIA。
- `aria-label`用于无可见文本的控件。
- `aria-describedby`用于关联辅助说明或错误信息。
- `aria-live`用于异步反馈，但避免过度播报。

### 16.4 焦点管理

- 打开Dialog、Drawer、Popover后，焦点移动到浮层内合理位置。
- 关闭浮层后，焦点返回触发元素。
- 删除列表项后，焦点移动到相邻项或列表容器。
- 页面跳转后，焦点应回到页面主标题或主内容区域。

### 16.5 可访问性检查清单

发布前检查：

-  页面可仅用键盘完成主要任务。
-  焦点顺序符合视觉和任务顺序。
-  焦点态清晰可见。
-  文本对比度达标。
-  表单字段有Label。
-  错误信息与字段关联。
-  图标按钮有可访问名称。
-  Dialog焦点管理正确。
-  动效遵守`prefers-reduced-motion`。
-  不仅依赖颜色传达状态。

## 17. Responsive响应式设计

### 17.1 断点

| Token            | 范围          | 说明         |
| ---------------- | ------------- | ------------ |
| `breakpoint.xs`  | <480px        | 小屏手机     |
| `breakpoint.sm`  | 480px-767px   | 常规手机     |
| `breakpoint.md`  | 768px-1023px  | 平板、小窗口 |
| `breakpoint.lg`  | 1024px-1439px | 常规桌面     |
| `breakpoint.xl`  | 1440px-1919px | 大屏桌面     |
| `breakpoint.2xl` | >=1920px      | 超宽屏       |

### 17.2 响应式规则

- 小屏优先单列布局。
- 表格在小屏上应转换为卡片、关键列展示或横向滚动。
- Sidebar在小屏上转为Drawer或底部导航。

- Hover信息必须有移动端替代方案。

## 18. Dark Mode深色模式

### 18.1 原则

- 深色模式使用独立alias token，不做简单反色。
- 降低大面积纯黑和纯白对比造成的眩光。
- 阴影在深色模式中可结合边框、背景层次表达。
- 状态色需要重新检查对比度。

### 18.2 实现要求

```css
:root {
  color-scheme: light;
}

[data-theme="dark"] {
  color-scheme: dark;
}
```

业务组件不得写死亮色或暗色值，应使用主题变量。

## 19. Content Design内容设计

### 19.1 文案原则

- 具体：说明用户正在做什么。
- 简短：去掉不影响理解的词。
- 可执行：错误和空状态应给下一步。
- 一致：同一概念使用同一名称。
- 中性：避免责备用户。

### 19.2 常见文案模板

#### 按钮

```text
创建项目
保存更改
取消
重试
查看详情
导出数据
删除项目
```

#### 空状态

```text
暂无[对象]
[创建/导入/添加][对象]后，你可以在这里[查看/管理/配置]。
```

#### 错误状态

```text
无法[完成动作]
[原因或可能原因]。请[修复动作]。
```

#### 删除确认

```text
删除[对象名称]？
此操作会永久删除[影响范围]，无法恢复。
```

### 19.3 禁用文案

避免：

- “出错了”
- “非法参数”
- “请正确填写”
- “操作失败”
- “未知错误”

除非用于兜底，且必须配合错误代码和重试/反馈入口。

## 20. Internationalization国际化

### 20.1 文本扩展

设计时预留文本扩展空间。英文、德文、俄文等语言可能比中文更长。

要求：

- 按钮允许适度变宽。
- 表格列支持截断和tooltip。
- 固定宽度容器需验证长文本。
- 不把文本嵌入图片。

### 20.2 RTL支持

若支持阿拉伯语、希伯来语等RTL语言：

- 布局方向镜像。
- 图标方向按语义判断是否镜像。
- 文本对齐遵循语言方向。
- 焦点顺序和键盘方向键需适配。

### 22.3 交付要求

设计交付必须包含：

- 页面状态：默认、加载、空、错误、权限不足。
- 响应式断点：至少桌面和移动端。
- 交互说明：点击、悬停、聚焦、展开、关闭。
- 文案：真实或接近真实数据，不用大量Lorem ipsum。
- Token标注：颜色、字号、间距、圆角、阴影尽量使用变量。

## 23. QA与验收清单

### 23.1 设计验收

-  页面主任务清晰。
-  页面标题、说明、操作层级正确。
-  使用既有组件，没有不必要的新样式。
-  颜色、字号、间距、圆角符合token。
-  所有状态完整：默认、加载、空、错误、禁用、成功。
-  表单校验和错误文案完整。
-  危险操作有确认或撤销机制。
-  小屏布局可用。
-  深色模式可用或有明确不支持说明。
-  可访问性检查通过。

### 23.2 开发验收

-  未硬编码设计值，优先使用token。
-  组件状态与设计一致。
-  键盘操作可用。
-  ARIA属性正确。
-  加载、错误、空状态实现完整。
-  表单校验不丢失输入。
-  异步提交防重复。
-  响应式断点符合设计。
-  单元测试或交互测试覆盖关键组件。
-  Storybook或等效组件预览已更新。

## 24. 反模式

禁止或避免以下做法：

- 为单个页面随意创建新颜色。
- 同一语义状态在不同页面使用不同颜色。
- 用Placeholder替代Label。
- 只用Toast展示关键错误。
- 删除不可恢复数据时不确认。
- 图标按钮没有可访问名称。
- 焦点态被CSS移除。
- 小屏只缩小不重排。
- 依赖颜色作为唯一状态表达。
- 在业务组件中硬编码十六进制颜色和阴影。
- 用复杂动画掩盖性能问题。
- 表格列过多但没有列管理或横向策略。

### 25.1 何时需要更新本文档

- 新增全局token。
- 修改品牌色、字体、圆角、阴影等基础规则。

- 修改关键交互模式。

## 27. 最低落地标准

若项目时间有限，至少必须做到：

1. 建立颜色、字号、间距、圆角token。
2. 按token实现Button、Input、Select、Table、Dialog、Toast、EmptyState、ErrorState。
3. 所有页面覆盖加载、空、错误状态。
4. 主要路径支持键盘操作。
5. 文本对比度满足要求。
6. 危险操作有确认或撤销。
7. 不在业务代码中硬编码设计值。
8. 设计变更同步更新本文档。
