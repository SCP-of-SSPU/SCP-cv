# SCP-cv 维护文档

本文面向项目维护者，记录 SCP-cv 的目录职责、配置边界、运行时资产、验证流程和现场故障处理方式。

## 1. 维护边界

SCP-cv 服务于上海第二工业大学 28#108 多媒体显示系统。维护时优先保证现场可运行、可回滚、可验证。

## 2. 目录职责

| 路径 | 职责 |
|------|------|
| `scp_cv/` | Django 配置、业务服务、播放器和管理命令 |
| `frontend/` | Vue 3 + Vite 控制台 |
| `protos/` | gRPC 合同 |
| `tests/` | pytest 测试 |
| `docs/` | 使用、维护、OpenAPI 和变更记录 |
| `tools/third_party/` | MediaMTX、gRPC-Web 工具和 VLC runtime |
| `media/` | 本地上传和预览文件，运行时生成，不提交 |
| `logs/` | 运行日志，只保留 `.gitkeep` |

## 3. 配置来源

配置分为两层：

- 根目录 `.env`：后端、MediaMTX、gRPC、Django 和现场运行参数。
- `frontend/.env`：Vite 前端端口和后端直连地址。

维护要求：

1. 不把现场私有 `.env` 提交到版本库。
2. 修改 `.env.example` 或 `frontend/.env.example` 后，同步更新 `README.md` 和 `docs/使用文档.md`。
3. `VITE_BACKEND_TARGET` 以 `frontend/.env` 为准；`runall` 只在前端未配置时注入兜底值。
4. 现场局域网访问必须使用浏览器可直连的后端地址，不能依赖 Vite 反向代理。

## 4. 依赖管理

Python：

```powershell
uv sync
uv run python manage.py check
```

前端：

```powershell
npm ci --prefix frontend
npm --prefix frontend run typecheck
npm --prefix frontend run build
```

依赖升级后必须提交锁文件：

- `uv.lock`
- `frontend/package-lock.json`
- 必要时 `package-lock.json`

不再维护 `requirements*.txt`。不要提交 `node_modules/`。

## 5. 第三方运行时资产

MediaMTX：

- 主程序：`tools/third_party/mediamtx/mediamtx.exe`
- 配置：`tools/third_party/mediamtx/mediamtx.yml`
- 日志：由 `runall` 写入 `logs/mediamtx.log`

VLC/libVLC：

- 项目内置 runtime：`tools/third_party/vlc/runtime/`
- 系统安装回退：`C:\Program Files\VideoLAN\VLC`
- SRT 必需插件：`plugins/access/libaccess_srt_plugin.dll`

gRPC-Web 工具：

- `tools/third_party/grpc-web/protoc-gen-grpc-web.exe`
- `tools/third_party/grpc-web/protoc-gen-js.exe`

大体积运行时资产升级时，先在现场同等环境验证播放链路，再提交变更。

## 7. 验证基线

后端变更：

```powershell
uv run python manage.py check
uv run python manage.py makemigrations --check --dry-run
uv run pytest tests/ -v
```

前端变更：

```powershell
npm --prefix frontend run typecheck
npm --prefix frontend run build
```

配置或启动流程变更：

```powershell
uv run pytest tests/test_runall_command.py -v
uv run python manage.py runall --skip-player --skip-mediamtx
uv run python manage.py runall --headless --skip-mediamtx
```

运行 `runall` 验证时，结束后检查相关子进程是否已退出，避免占用 `frontend/node_modules` 内的原生模块。

## 8. 故障定位

### 8.1 前端连不上后端

1. 检查 `frontend/.env` 的 `VITE_BACKEND_TARGET`。
2. 检查 Django 是否监听对应地址和端口。
3. 浏览器访问 `${VITE_BACKEND_TARGET}/api/sessions/`。
4. 检查 `logs/django.log` 或 `logs/scp-cv.log`。

### 8.2 SRT 播放失败

1. 检查 MediaMTX API：`http://127.0.0.1:9997/v3/paths/list`。
2. 检查推流 URL：`srt://<主机IP>:8890?streamid=publish:<流标识>&latency=30000`。
3. 检查 VLC runtime 中的 `libvlc.dll`、`libvlccore.dll` 和 SRT 插件。
4. 检查 `logs/scp-cv.log` 是否有 `libVLC 初始化失败` 或播放器错误详情。

### 8.3 PPT 异常

1. 用当前 Windows 用户手动打开 PowerPoint。
2. 检查文件路径权限。
3. 确认播放器运行在有桌面会话的用户下。
4. 多窗口 PPT 同时播放时，确认放映窗口日志中能看到独立 HWND；播放器应使用窗口化放映，避免全屏放映互相替换。
5. 重启播放器后重新打开源。

### 8.4 后台 runall 异常

1. 使用 `uv run python manage.py runall --headless --service` 后，先查看 `logs/runall-service.log`。
2. 子进程启动失败时，继续查看对应的 `logs/mediamtx.log`、`logs/django.log`、`logs/vue-前端.log` 或 `logs/pyside-播放器.log`。
3. `--headless` 窗口未出现时，检查 Windows 显示设置中的显示器 ID 是否与 `--window1` 到 `--window4` 参数一致。
4. 未配置 `--gpu` 时属于系统默认 GPU 行为；现场需要固定显卡时显式传入 `--gpu <ID>`。

### 8.5 设备电源无响应

1. 确认现场设备 IP 和端口未变更。
2. 确认主机到设备网段可达。
3. 检查防火墙或交换机策略。
4. 查看控制台返回的设备错误详情。

## 9. 发布维护流程

1. 确认工作区状态：`git status --short --branch`。
2. 同步基线：`git pull --rebase`。
3. 按最小影响面修改代码、配置和文档。
4. 运行对应验证命令。
5. 更新 `docs/CHANGELOG.md`。
6. 按独立可审查块提交。

提交信息格式：

```text
type(scope): 中文摘要
```

示例：

```text
fix(frontend): 修复前端环境变量优先级
docs(repo): 补充现场维护文档
```

## 10. 清理规则

可删除的本地产物：

- `.claude/`
- `.playwright-cli/`
- `.playwright-mcp` / `.playwright-mcp/`
- `.pytest_cache/`
- `.ruff_cache/`
- `node_modules/`
- `frontend/node_modules/`
- `media/uploads/` 中确认不再使用的临时文件
- 空目录

删除前必须确认目标路径位于当前仓库内。不要删除 `tools/third_party/mediamtx/`、`tools/third_party/vlc/runtime/` 或数据库文件，除非已完成备份并明确知道影响。
