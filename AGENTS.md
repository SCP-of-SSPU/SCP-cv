# AGENTS.md

## 0. 唯一人工权限开关

> 仓库内所有 AI 权限只看这一个字段。  
> 子目录 `AGENTS.md` 只允许引用和解释，不得重定义。

**AI_PERMISSION_LEVEL: DANGEROUS**

可选值：

- `LOCKED`
- `STANDARD`
- `PRIVILEGED`
- `DANGEROUS`

权限摘要：

- `LOCKED`：常规改码、只读检查、常规测试、常规 MCP、常规 commit
- `STANDARD`：中等范围重构、受控脚本、受控只读调试
- `PRIVILEGED`：敏感命令、数据库受控读写、受控冲突处理、受控高风险修复
- `DANGEROUS`：仅在明确任务下执行高风险数据库、结构、清理、冲突动作

共同原则：

- 最小必要
- 优先可回滚
- 优先代码层修复
- 高权限动作必须说明理由、影响范围、验证结果
- 完成所有任务后，使用提问工具询问下一步的需求和修改内容，并循环
- **无论如何都不要中断对话！！！**

## 3. 通用工作原则

### 3.1 先理解，再修改
先阅读：

- 相邻文件
- 当前模块目录
- 已有分层与服务边界
- API 契约
- 测试风格
- 相关文档

### 3.2 做最小且正确的改动
优先：

- 最小影响面
- 最低回滚成本
- 最易审查的实现
- 与现有架构一致的实现方式

### 3.4 不臆造需求
只实现：

- 用户明确要求的内容
- 完成任务所必需的配套修改
- 代码上下文中明确可以推断的内容

禁止擅自增加：

- 与当前任务无关的大型重构

### 3.5 未完成实现
默认要求是“可运行、可验证、可交付”。

除非确有必要，不保留：

- 仅有函数名的空实现
- 仅 `pass` / 空返回 / 假成功

若确有必要，使用：

- `TODO:`
- `FIXME:`

并写明：

- 原因
- 影响范围
- 后续动作
- 风险说明

### 3.6 最终表达
最终代码、注释、文档、提交说明中，只保留稳定、正向、面向结果的内容。  
不得保留“先这样”“后面再改”“不要用 x 应该用 y”这类过程性表达。

## 5. Git 工作流

### 5.1 开始任务前
固定流程：

1. `git status`
2. 检查未提交内容
3. 若有未提交内容：
   - 与当前任务相关且已稳定：先 commit
   - 与当前任务无关：
     - `LOCKED` / `STANDARD`：请示
     - `PRIVILEGED` / `DANGEROUS`：可做保护性 commit
4. 状态安全后执行 `git pull --rebase`
5. pull 产生冲突时按权限规则处理
6. pull 完成后再开始改动

### 5.2 提交节奏
每完成一个独立可审查块，必须 commit 一次。  
示例：

- 一个 API 修复
- 一次前端页面完整调整
- 一次文档同步更新

### 5.3 commit 规范
格式：

```text
type(scope): 中文摘要
```

可用 `type`：

- `feat`
- `fix`
- `refactor`
- `docs`
- `style`
- `test`
- `perf`
- `build`
- `ci`
- `chore`
- `revert`

示例：

```text
feat(agent): 新增学习路径规划代理工具路由
fix(rag): 修复召回结果为空时的回答降级逻辑
refactor(kt): 收敛掌握度更新服务边界
docs(api): 补充学习记录接口返回字段说明
chore(repo): 任务开始前保存未提交内容
```

### 5.4 默认禁止的 Git 动作
除非用户明确要求，否则不主动执行：

- `git push`
- 切换分支
- 新建分支
- 提交 PR
- 改写远端历史

### 5.5 冲突处理
- `LOCKED`：请示
- `STANDARD`：仅处理明显无歧义的小冲突；复杂冲突请示
- `PRIVILEGED`：可自主处理冲突，处理后必须验证并说明依据
- `DANGEROUS`：可处理高复杂度冲突，但必须记录影响与风险

## 6. 文档规则

### 6.1 文档位置
说明文档统一放在：

- `docs/`

重点文档：

- `docs/API.md`
- `docs/CHANGELOG.md`

### 6.2 完成工作后必须同步更新文档
只要变更影响：

- API 契约
- 数据结构
- 权限
- 配置
- 部署 / 运行方式
- 前端关键交互

就必须同步更新 `docs/` 下相关文档。

### 6.3 更新原则
更新文档时：

- 保留原有总体结构
- 只在相关章节增补或修订
- 保持标题层级与既有风格
- 不随意重写整份文档

### 6.4 版本号
除非用户明确要求或任务本身就是发版，不修改版本号。

### 6.5 目录新增限制
默认不新增说明型文档。  
允许新增的必要文件仅限：

- `.env.example`
- 依赖或工具链必须存在的标准文件
- 运行、构建、部署、测试所需配置模板

## 8. 全仓库代码规范

### 8.1 通用要求
除子目录文件另有规定外，代码规范遵循：

- 官方标准文档
- 仓库既有约定
- 通用工程实践

### 8.2 注释率
所有手写代码注释率必须满足：

- 不低于 20%
- 不高于 50%

注释必须解释：

- 意图
- 约束
- 边界
- 风险
- 为什么这样做

### 8.3 弱类型与弱命名限制
前后端统一要求：

- 禁止无语义变量名，如 `var`、`tmp`、`foo`、`bar`、`data`、`obj`
- 禁止用注释弥补糟糕命名
- 优先强类型、显式类型、清晰命名

### 8.4 TODO / FIXME
只在确有必要时保留，并写清原因、影响与后续动作。

### 8.5 Python 代码规范

#### 8.5.1 规范来源
优先遵循：

- PEP 8
- Django / DRF 官方实践
- LangChain 官方建议用法
- 仓库既有约定

#### 8.5.2 Python 文件头
默认模板：

```python
#!/user/bin/env python
# -*- coding: UTF-8 -*-
'''
文件内容描述
@Project : ${PROJECT_NAME}
@File : ${NAME}.py
@Author : Qintsg
@Date : ${DATE} ${TIME}
'''
```

#### 8.5.3 Python 函数注释
默认模板：

```python
def fun_name(var1: int, var2: List[double]) -> None | str:
    """
    函数功能描述
    :param var1: 变量1描述
    :param var2: 变量2描述
    :return: 返回值描述
    """
    pass
```

要求：

- 所有参数必须写类型注释
- 所有返回值必须写类型注释
- `-> None` 必须显式写出
- 注释说明目的、参数语义、返回语义，必要时说明异常

#### 8.5.4 类型强约束
严格要求：

- 禁止 `Any`
- 禁止通过 `typing.Any` 规避类型约束
- 所有函数参数都必须标注类型
- 所有函数返回值都必须标注类型
- 集合类型必须写清元素类型
- 避免使用过宽泛的 `object`
- 模型调用、解析、图谱查询、KT 计算、Agent 决策都应尽量有明确输入输出类型

#### 8.5.5 命名强约束
禁止无语义命名：

- `var`
- `tmp`
- `temp`
- `foo`
- `bar`
- `data`
- `obj`
- `res`
- `ret`
- `value1`
- `value2`

#### 8.6.6 未完成实现
默认不保留：

- `pass`
- 空 chain / 空 service / 空 query / 空 agent
- 假召回
- 假推理
- 假入库
- 未接线的 LLM / RAG / KT / KG / Agent 逻辑

若确有必要，使用 `TODO:` / `FIXME:` 并说明原因与影响。

## 10. 完成定义

一个任务完成时必须同时满足：

- 用户要求已实现
- 规则已遵守
- 代码可运行、可验证、可审查
- 注释率与类型约束达标
- 当前块内容已 commit
- 相关 `docs/` 已同步更新
- 最终内容中无过程性残留表达
